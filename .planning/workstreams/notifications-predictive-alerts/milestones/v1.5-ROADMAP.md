# Milestone v1.5: TUI Dashboard

**Status:** ✅ SHIPPED 2026-07-24
**Phases:** 8-9
**Total Plans:** 4

## Overview

Surface the tray's usage/quota/trends and live sessions in a terminal UI, so
people who live in the terminal get the same data without a browser round-trip.
Closes SEED-007. A new `claude-tui.py` entry point becomes a third consumer of
`claude_monitor.core` alongside `claude-monitor.py` and `dashboard.py`, fed by a
new read-only query verb on the daemon's existing unix socket.

v1.5 takes the project's first exception to the stdlib+PyGObject-only rule —
`textual` as a runtime dependency, scoped to the one new entry point via a PEP 723
inline block and an optional `tui` extra, so the daemon's system interpreter never
gains it.

## Phases

### Phase 8: Daemon Socket Query Verb

**Goal**: The daemon's existing unix socket can answer a read-only query for the live session table plus the latest usage/history state, without disrupting or blocking the existing fire-and-forget hook-event path.
**Depends on**: Phase 7 (`self.sessions` and `write_dashboard`'s read-only snapshot precedent)
**Requirements**: SOCK-01, SOCK-02, SOCK-03
**Plans**: 2 plans

Plans:

- [x] 08-01: sessions_lock + core.build_session_snapshot (thread-safety foundation, SOCK-03 + SOCK-01 shape groundwork)
- [x] 08-02: thread-per-connection serve() + query dispatch + socket 0600 hardening (SOCK-01/02/03 live)

**Details:**
Success criteria met: a `{"query": "snapshot"}` message returns JSON with every
tracked session (dir/status/pane/tmux) plus the last polled usage/history; hook
events keep flowing un-slowed alongside query connections; a stalled/malformed
query connection is contained to its own daemon thread (`_handle_conn`,
`threading.Thread(target=..., daemon=True)`); the snapshot is built under
`mon.sessions_lock` so a read racing a Gtk-thread mutation returns the before- or
after-state, never a torn one. `mon.usage`/`mon.trends` are read outside the lock
as single-reference rebinds, matching `compute_trends`' existing no-lock posture.

### Phase 9: Terminal Dashboard (claude-tui.py)

**Goal**: A person living in a terminal sees the tray's full usage/quota/trends/live-sessions picture without a browser — a new `claude-tui.py` entry point querying Phase 8's socket verb.
**Depends on**: Phase 8 (the socket query verb is this phase's only data source)
**Requirements**: TUI-01, TUI-02, TUI-03, TUI-04, TUI-05
**Plans**: 2 plans

Plans:

- [x] 09-01: pure TUI substrate in `claude_monitor/core.py` (socket client, usage rows, trend text, session rows, timing constants) plus its `--selfcheck` asserts, all above the textual boundary
- [x] 09-02: `claude-tui.py` textual App, CSS and two-timer refresh loop, plus packaging (`pyproject.toml` `tui` extra, `install.sh` symlink, `just tui`)

**Details:**
Success criteria met: `claude-tui.py` shows current usage (%, tokens, reset
countdown, burn rate) for both the 5h and 7d caps; trends (sparkline, daily/weekly
burn, peak-usage hour) reuse `core`'s existing trend functions verbatim rather than
recomputing; a live sessions panel lists every tracked session sorted
waiting -> running -> done matching the v1.4 dashboard semantics; the screen
auto-refreshes on an interval; an unreachable daemon shows a clear message, never a
crash or traceback. Post-UAT hardening (WR-01..06, CR-01/02) bounded `read_line`
with a wall-clock deadline + size cap, rejected non-object snapshot JSON, stripped
control chars from session-dir cells (ANSI-injection guard), preserved sessions
scroll position across the 1s render tick, and read usage keys defensively.

## Milestone Summary

**Decimal Phases:** None.

**Key Decisions:**

- **v1.5 is 2 phases** (8: daemon socket query verb; 9: the TUI renderer). Phase 9 has no data source until Phase 8's query verb exists and is independently verifiable (connect + query the socket, no TUI needed) — a real unblocks-the-next-phase boundary, not an arbitrary horizontal-layer split.
- **Shared-socket query verb over standalone history-file read.** Chosen to get live sessions into v1.5 scope; a no-daemon fallback reading `usage-history.jsonl` directly is deferred.
- **The whole TUI substrate lives in `core.py` above the textual boundary**, so `--selfcheck` on `/usr/bin/python3` proves it; `claude-tui.py` is App-class-and-CSS only. `query_snapshot` raises on every failure mode; degraded-mode swallowing lives at the App's worker boundary.
- **`textual` ships as an OPTIONAL `tui` extra** (default `uv sync` stays at zero third-party packages); runtime resolution is `claude-tui.py`'s own PEP 723 block, so a plain `./claude-tui.py` works from any cwd and the daemon's PEP 668 interpreter never gains textual.
- **Double containment on every textual callback** — `@work(exit_on_error=False)` plus a blanket `except Exception` in the body — because `App._handle_exception` always exits.
- **`serve()` refactored to thread-per-connection** (`_handle_conn`) so a stalled/malformed connection blocks only its own thread; query snapshot build runs entirely inside `mon.sessions_lock`.

**Issues Resolved:**

- SOCK-03 torn-read risk closed with `sessions_lock` wrapping every `self.sessions` call site as one atomic read-modify-write / snapshot-build.
- Post-UAT review fixes: unbounded socket read (WR-01), non-object JSON at parse boundary (WR-02), render-failure crash path routed through the tick guard (WR-03), command-palette leak so `q` is the only binding (WR-04), unpinned tui deps (WR-05, committed per-script uv lockfile), missing usage keys (WR-06), ANSI injection via session dir cells (CR-02), lost scroll position on render tick (CR-01).

**Issues Deferred:**

- Click-to-focus a pane from the TUI (reusing `pane`/`tmux` fields) — the tray remains the click-to-focus surface for v1.5.
- Standalone (no-daemon) mode reading `usage-history.jsonl` directly.
- TUI polish (color-by-threshold, progress bars, richer trends, optional pane focus) — proposed for a v1.6 milestone (TUI feels v1.0-tier).

**Technical Debt Incurred:**

- No `v1.5-MILESTONE-AUDIT.md` — `/gsd-audit-milestone` was not run for v1.5; close proceeded on 8/8 requirements complete + Phase 9 UAT passed (3 pass, 2 skipped, 0 issues).
- Phase 9 UAT skipped 2 live-observation checks (scroll-retention CR-01, daemon-outage D5/WR-03), documented as non-failures verified by tests + code re-read.
