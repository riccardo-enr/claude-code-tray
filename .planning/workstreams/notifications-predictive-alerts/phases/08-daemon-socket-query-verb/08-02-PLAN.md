<objective>
Turn serve()'s single-threaded accept loop into thread-per-connection (D-02/D-03), then add the
`{"query": "snapshot"}` dispatch branch that answers with the shared sessions+usage+trends payload
from Plan 08-01, and harden the socket file's permissions now that it answers reads, not just
accepts writes.

Purpose: `serve()` today has no way to answer a query without either blocking the accept loop
(stalling every subsequent hook event behind a slow/malformed client) or adding a bounded timeout
that still leaves a window of stall. Thread-per-connection removes the stall entirely -- a
malformed or slow query connection only ever affects its own thread (SOCK-02). Once connections
are isolated, the query verb itself (SOCK-01) is a small addition: recognize `"query"` in the
message, build the shared snapshot under Plan 08-01's lock (SOCK-03), and reply on the same
connection before closing it (D-05).
Output: `_handle_conn`, thread-per-connection `serve()`, the `"query"` dispatch branch, and a
`chmod(SOCK, 0o600)` hardening line.
</objective>

## Prohibition Recall (Stage 1 recall -> Stage 2 precision)

Ran the adversarial recall pass ("What could this feature silently become that the author would
NOT want, but the spec does not forbid?") against SOCK-01/02/03. Stage 1 surfaced ~10 raw
candidates (write/mutation creep, data leakage beyond D-06's scope, network-exposure creep,
indefinitely-stale trends with no freshness signal, unbounded thread/connection growth, a torn
read crashing a connection thread, malformed-payload memory exhaustion, lock re-entrancy deadlock,
a "read" with notification side effects, and the `/tmp` socket-permissions fallback). Stage 2
precision dropped every candidate: four are routine-engineering or pre-existing characteristics of
the socket unchanged by this phase (data scope is already bounded by D-06; stale trends already
degrade the same way today; malformed-payload exposure already exists for hook events; lock
re-entrancy is a code-review-level correctness concern addressed by this plan's flat lock-acquire
shape); one folds into the SOCK-03 idempotency edge already captured in `must_haves.truths`
(side-effect-free query); and three are canon security items referred rather than minted:

- Write/mutation creep -> canon-referral: broken access control / insecure-design canon, covered
  by `/gsd-secure-phase` and this plan's own explicit scope (only `"query": "snapshot"` is
  recognized; no other query value does anything); not minted as a bespoke prohibition.
- Unbounded connection/thread growth -> canon-referral: OWASP unrestricted-resource-consumption
  canon; addressed instead via the STRIDE register below (T-08-03, accepted given the local-only
  trust boundary) rather than a bespoke prohibition.
- `/tmp` socket-permissions fallback (a local other-user reading your session/usage data) ->
  canon-referral: broken access control / information-disclosure canon; addressed instead via an
  actual code mitigation in Task 2 (`os.chmod(SOCK, 0o600)`) and the STRIDE register below (T-08-04)
  rather than a bespoke prohibition.

Zero bespoke prohibitions survived Stage 2 -- the correct precision outcome for a backend-only,
read-only, local-IPC protocol change with no user-facing product/values/fairness surface (see the
prohibition-probe "clean-utility" worked example). `must_haves.prohibitions` is therefore omitted
from this plan's frontmatter rather than left as a dishonest empty placeholder.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local process (any user-owned client) -> unix socket -> `_handle_conn` | The socket accepts connections from any local process that can open the socket file. This phase adds a READ path (previously the socket only accepted fire-and-forget writes); the trust boundary and its consequences are reassessed accordingly. Per the security contribution for this phase: local IPC, not network-exposed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08-03 | Denial of Service (unbounded thread spawn) | `serve()`'s `threading.Thread` per accepted connection | low | accept | A local client opening many connections without closing spawns one daemon thread per connection; daemon threads are cheap and die with the process. Requires local code execution as the SAME user (or another local user, mitigated by T-08-04's chmod) to exploit -- an attacker with that access already has far larger attack surface. Matches the existing accepted-risk posture for hook events (also unrate-limited). No throttling added; canon-referred (OWASP unrestricted-resource-consumption) rather than bespoke-mitigated. |
| T-08-04 | Information Disclosure | `SOCK`'s `/tmp` fallback when `XDG_RUNTIME_DIR` is unset | medium | mitigate | Before this phase, another local user reaching the `/tmp`-fallback socket could only forge fake hook events (annoying, low-impact). The query verb elevates that to reading live project directory names and usage% (information disclosure). Mitigated by `os.chmod(SOCK, 0o600)` immediately after `srv.bind(SOCK)` (Task 2), restricting connect access to the owning user regardless of the parent directory's mode. |
| T-08-05 | Tampering (malformed/adversarial query JSON) | `_handle_conn`'s per-line `json.loads` + query dispatch | low | accept | Unchanged from the existing hook-event precedent: a bad JSON line is silently skipped via the existing per-line `try/except Exception: continue`, and the outer per-connection `except Exception: traceback.print_exc()` (Task 1) isolates any other failure to one connection's thread. |

No package-manager installs in this plan (stdlib only) -- no supply-chain (T-08-SC) threat applies.
