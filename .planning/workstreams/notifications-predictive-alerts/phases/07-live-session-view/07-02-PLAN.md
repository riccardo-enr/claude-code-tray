## Root cause (from `.planning/debug/stale-session-status-stuck.md`)

`self.sessions`'s ONLY removal path is the `end` socket event, fired exclusively by Claude
Code's `SessionEnd` hook. That hook does not fire on `/exit` (anthropics/claude-code#17885) or
`/clear` (anthropics/claude-code#6428), and cannot fire at all when the pane/process is killed
externally. When `end` never arrives, the entry is frozen forever at whatever status it last
received, and both `rebuild_menu()` (tray) and `write_dashboard()` (dashboard panel, added in
07-01) snapshot it verbatim every tick -- including the dashboard's 1s client-side duration
ticker, which has no liveness signal of its own and just counts `now - entered` up forever.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| poll thread -> Gtk thread (`self.sessions` mutation) | `reap_stale` runs on the poll thread and must never mutate `self.sessions` directly; the pop is handed back to the Gtk thread via `GLib.idle_add`, the same boundary 07-01's T-07-02 already documented for the read-only snapshot in `write_dashboard`. This plan adds a WRITE path from that boundary for the first time (previously only reads crossed it), which is exactly why the single-mutator invariant matters more now. |
| `pane`/`tmux` values -> `subprocess.run(["tmux", ...])` | Same trust level as the existing `focus()` and `pane_onscreen()` calls -- these values originate from `TMUX_PANE`/`TMUX` env vars of the local, same-user hook-invoking shell (via `claude-send.py`), not from any external/network input. `pane_alive` reuses this already-trusted local-IPC path; no new attacker surface. |
