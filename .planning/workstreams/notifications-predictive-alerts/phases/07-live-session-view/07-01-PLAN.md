<objective>
Embed the tray's in-memory session list into the existing self-contained dashboard as a
live-refreshing Sessions panel: project dir, status (colored dot + word), and a
client-ticked time-in-state, sorted waiting -> running -> done with done rows dimmed, and
a clean empty state. Read-side only -- no new IPC, socket, or persistence; the panel is a
pure function of the in-memory snapshot at generation time and rides the existing
meta-refresh cadence (only the duration counter is live via inline JS).

Purpose: see every live Claude Code session at a glance in the dashboard, not just in the
tray menu (v1.4 milestone goal, SESSVIEW-01..05).
Output: an `entered` session field, a snapshot threaded into render_dashboard, a
`"sessions"` payload key, inline panel HTML/CSS/JS, and locking --selfcheck asserts.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| session `dir` -> dashboard HTML | The project directory name originates from an arbitrary repo path (Claude Code hook `cwd`). It is untrusted string data crossing into the self-contained `file://` page, which the user opens in a browser. Same class of hazard as notifier threat T-05-04. |
| Gtk main thread -> poll thread | `self.sessions` is mutated in `Monitor.handle()` on the Gtk thread and read (snapshotted) in `write_dashboard()` on the poll thread. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-01 | Tampering / Info-disclosure (XSS) | render_dashboard sessions -> dashboard page | medium | mitigate | Session data ships ONLY inside the `_embed_json` payload (escapes `< > &` to \u-escapes, breaking `</script>` and tags) AND rows are built client-side with DOM `textContent`, never HTML string interpolation of `dir`. Asserted in --selfcheck: a dir of angle-bracket markup never appears as raw markup and `</script>` count stays 1 (Task 3b). |
| T-07-02 | Denial of Service (thread race) | Monitor.write_dashboard snapshot | low | accept | `self.sessions` may be mutated mid-snapshot from the Gtk thread. The existing `except Exception: return` in write_dashboard degrades any race to one skipped regen tick -- no crash, no dead thread (matches NOTIF-04 / POLL-02 posture). No lock added; per-session locking is the upgrade path only if skipped ticks are observed. |

No package-manager installs in this phase (stdlib + PyGObject only, already present) -- no supply-chain (T-07-SC) threat applies.
</threat_model>
