<objective>
Ride the shared path with the first producer: session `waiting` / `done` notifications,
fed by the existing hook -> unix socket -> `GLib.idle_add` -> `Monitor.handle` pipeline.

No new polling, no new data source, no new socket. The de-dupe is a pure comparison of
old status against new -- assertable in `--selfcheck` without a bus.

Purpose: SESS-01 / SESS-02, and the first proof that `emit_notif` generalizes.
Output: `sess_should_notify()` + one call into `Monitor.emit_notif` from `Monitor.handle`.
</objective>

  <action>
    Add module-level `sess_should_notify(old_status, new_status)`. Pure -- no `self`, no I/O,
    no clock. It returns True iff `new_status` is one of `"waiting"` / `"done"` AND
    `old_status != new_status`.

    That single expression carries NOTIF-02's de-dupe entirely: a session sitting in `waiting`
    across many hook events re-sends `waiting` every time, and only a *change* passes. It also
    correctly notifies a brand-new session that arrives already `waiting` (`old_status` is

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| hook payload -> `Monitor.handle` -> notification body/title | `cwd` comes from the Claude Code hook, i.e. from whatever directory the user has checked out. A repository can be named anything, including a string of markup. It reaches the notification as `d`. Untrusted input crossing into a renderer. |
| session D-Bus -> `on_notif_signal` -> `Monitor.focus()` | This plan supplies the first action tuple that the (already-filtered) click dispatcher can invoke. The filter itself is Plan 01's T-05-01. |

## STRIDE Threat Register (ASVS L1, block on high)

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-04 | Tampering | notification body, `Monitor.handle` emit | **high** | mitigate | The daemon parses the notification **body** as Pango markup unconditionally (`useBodyMarkup: true`) and advertises `body-markup` in `GetCapabilities`. The session `dir` is derived from the hook's `cwd` and is attacker-influenceable (a repo named with markup characters). Mitigation: `dir` is passed as the `title` (summary), which this daemon does **not** markup-parse, and the `body` is one of two fixed literal ASCII strings with zero interpolation (Task 2). If any interpolated value ever lands in a body in a later phase, it MUST go through `GLib.markup_escape_text()`. **Blocking: the title/body split is not cosmetic, it is the control.** |
| T-05-05 | Elevation of Privilege | `("focus", pane, tmux)` action tuple | medium | mitigate | The action tuple is stored against a daemon-issued id and invoked only when `on_notif_signal` finds that id in `self.notif_acts` (Plan 01, T-05-01). `pane` / `tmux` originate from our own hook payload and are already passed to `subprocess.run` as an argv list (no shell) by the existing `Monitor.focus()` at :1450 -- unchanged by this plan, and the same path the existing menu click already takes. |
| T-05-06 | Spoofing | hook payload `session_id` | low | accept | Any local process can already forge a session event over the unix socket -- a pre-existing boundary this milestone does not widen. The worst outcome is a spurious notification whose click focuses a pane the payload itself named. Same trust level as the existing tray row. |
| T-05-SC | Tampering | supply chain | n/a | accept | No package installs in this phase. |
</threat_model>
