  <action>
    Wrap `serve()`'s **per-connection body** -- everything after `conn, _ = srv.accept()` -- in
    a `try` / `except Exception: traceback.print_exc(); continue`. `traceback` is already
    imported at :23.

    Placement is the entire point and the only thing that can be got wrong here:
    - `srv.accept()` stays **OUTSIDE** the `try`. A raise there is the listening socket itself
      dying, which is unrecoverable and has nothing useful to retry.
    - The guard goes **INSIDE** the `while`, not around it. Around the `while`, one bad
      connection still kills the thread: the `except` catches, control falls out of the loop,
      and the function returns -- functionally identical to today.
    - The existing `try: buf = conn.recv(...)` / `finally: conn.close()` pair is **preserved,
      nested inside** the new guard. Wrapping the body naively and dropping that `finally`
      leaks a file descriptor on every `recv` failure.

    Make the degradation observable, not silent, exactly as `poll_loop` does: a persistent
    failure prints a full traceback to stderr (the journal) on every failing connection; a
    transient one costs one connection.

    Add a `ponytail:` docstring on `serve()` naming why this is load-bearing: this thread is
    the ONLY thing feeding session events, so any raise escaping one connection kills it
    permanently and the tray goes silently deaf to every session forever -- the same failure
    shape as the corrupt-record bug (quick task `260713-fry`), and what NOTIF-04 is actually
    protecting. Note that `Monitor.handle` is about to become a new source of raises (Plan 02
    wires the SESS emit into it) but runs on the Gtk main thread via `idle_add`, so a raise
    there kills the idle source, not this thread.

    Do not otherwise change `serve()`'s behavior: the `_onscreen` pre-ack computation at
    :1721-1722 stays exactly as it is (D-04 -- that signal keeps gating only the `!` badge,

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| session D-Bus -> our process | `ActionInvoked` / `NotificationClosed` are **broadcast** signals on `/org/freedesktop/Notifications`. Every application's notification clicks -- and anything else able to emit on the session bus -- cross into `on_notif_signal`. Untrusted input. |
| unix socket -> `serve()` | Any local process can connect to `$XDG_RUNTIME_DIR/claude-monitor.sock` and write arbitrary bytes. Pre-existing boundary; this plan hardens the thread behind it. |
| notification daemon -> our process | The `Notify` reply id is daemon-supplied, used as a dict key only. Low value. |

## STRIDE Threat Register (ASVS L1, block on high)

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-01 | Spoofing / Elevation of Privilege | `Monitor.on_notif_signal` | **high** | mitigate | `ActionInvoked` is a broadcast. The handler MUST look `params[0]` up in `self.notif_acts` and `return` when absent (Task 2). Without the filter, any application's notification click -- or any process that can emit on the session bus -- drives `Monitor.focus()` (`tmux select-pane` + `wmctrl -x -a`) or `open_dashboard()`. That is an unauthenticated caller steering our window-focus action. **Blocking: must be mitigated before this plan closes.** |
| T-05-02 | Denial of Service | `serve()` socket thread | **high** | mitigate | An unguarded raise on any connection kills the socket thread permanently and the tray goes deaf to every session forever. Any local process can trigger it with one crafted frame. Mitigated by Task 3's per-connection guard. **Blocking.** |
| T-05-03 | Tampering | `emit_notif` `body` argument | medium | mitigate | The notification body is parsed as Pango markup by this daemon (`useBodyMarkup: true`, unconditional). This plan ships `emit_notif` with no producer; the constraint is enforced where the strings are built -- see Plan 02 (T-05-04). `emit_notif` itself interpolates nothing. |
| T-05-04 | Information Disclosure | `traceback.print_exc()` in `serve()` | low | accept | Tracebacks go to stderr / the systemd journal of a single-user local helper. Same posture `poll_loop` already ships (`260713-fry`). Making the failure observable is worth more than hiding a local stack trace. |
| T-05-SC | Tampering | supply chain | n/a | accept | **No package installs in this phase.** `Gio` ships with the already-present PyGObject; RESEARCH.md's Package Legitimacy Audit records "not applicable -- nothing fetched from a registry". No legitimacy checkpoint is required. |
