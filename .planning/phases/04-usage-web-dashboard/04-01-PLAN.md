<objective>
Extend the single-file tray helper `claude-monitor.py` so a new "Open Usage Dashboard"
tray menu item opens a self-contained HTML page (a static `.html` opened via `file://`,
per D-01) that renders the persisted `~/.claude/usage-history.jsonl` as three real charts:
a selectable-range (day/week/all) usage-% trend, a daily burn-rate trend over the full
history, and an hour-of-day x day-of-week peak-usage heatmap. The page embeds the history
as JSON plus small inline JS that draws the charts client-side (D-02), with range switching
done by client-side filtering of the one embedded dataset (D-03). The HTML is regenerated
in the background `poll_loop` off the Gtk main thread on a ~5min throttle (D-04); the menu
item only opens the already-written file (zero I/O, D-05). Heatmap cells use mean burn
rate `mean(burn)*60` converted once (D-06) on a single-hue ramp with gray empty cells (D-07).

Purpose: complement (not replace) the cramped tray trend rows with a browsable page over
longer ranges than the tray sparkline holds — read-only, stdlib-only, no new poll, no new
dependency, no second data source.

Output: `claude-monitor.py` extended in place with new pure functions
(`render_dashboard`, `heatmap_buckets`, `burn_series`, `_embed_json`), new constants
(`DASH_DIR`/`DASH_PATH`/`DASH_INTERVAL`), a `Monitor.write_dashboard` generation step wired
into `poll_loop`, a sensitive "Open Usage Dashboard" menu item with an `open_dashboard`
handler, and `demo()`/`--selfcheck` asserts for the new logic.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| history JSONL -> dashboard generator | `~/.claude/usage-history.jsonl` is loaded by `parse_history`, which validates only numeric `t`; `pct`/`burn` values are untrusted (a corrupt or externally-tampered line can carry arbitrary strings) and get embedded into an inline `<script>`. |
| generated HTML -> browser | The `file://` page executes its own inline JS in the user's browser under a local-file origin; no network egress, no external assets. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-01 | Tampering/Injection | `render_dashboard` embedding into inline `<script>` | high | mitigate | Serialize the embedded payload ONLY through `_embed_json`, which escapes `<` `>` `&` to their JSON unicode escapes so no history value (even a crafted script-closing sequence) can terminate the script element. A `demo()` assert renders a crafted string-valued `pct` and confirms the raw sequence is absent. |
| T-04-02 | Information disclosure | `dashboard.html` written under the user cache dir | low | accept | Output lives under `${XDG_CACHE_HOME:-~/.cache}/claude-tray/` (user-owned, single-user machine) and contains only usage-% and burn figures — no tokens, credentials, or message content — consistent with the JSONL already stored at `~/.claude/`. No secrets to leak. |
| T-04-03 | Denial of service | `Monitor.write_dashboard` on the poll thread | medium | mitigate | Generation runs off the Gtk main thread on a ~5min throttle; the input is already bounded by HIST-02 retention pruning; `write_dashboard` wraps its whole body in a broad `except Exception -> return` so a huge or malformed record degrades to "not updated this tick" instead of killing the poll thread or freezing the tray. |
| T-04-SC | Tampering | dependency supply chain | low | accept | No package installs this phase — stdlib only (`json`, `os`, `tempfile`, `datetime`, `webbrowser`) plus the existing PyGObject. Supply-chain surface unchanged; nothing to audit. |
</threat_model>
